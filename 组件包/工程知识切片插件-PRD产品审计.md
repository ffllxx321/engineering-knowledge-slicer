# 工程知识切片插件 PRD 产品审计

## 结论

v1.0 已经确定正确的产品方向，但还不能单独指导当前插件重构。问题不在“功能点少”，而在若干关键行为没有唯一答案，开发者仍需自行猜测。v1.1 已补齐阻塞性产品决策，可以进入工程契约和插件重构阶段。

## P0 阻塞项

| 缺口 | 风险 | v1.1 处理 |
|---|---|---|
| “两次 MiniMax 调用”与长文档分段矛盾 | 长文档遗漏、重复、跨段结论丢失 | 改为两类逻辑职责，规定章节 map/reduce、覆盖率和全局去重 |
| 支持格式与 MinerU/PaddleOCR 能力未对齐 | EML、Office、HTML、图片实现各自猜测 | 增加格式/解析器矩阵和 API 限制 |
| 没有机器可校验数据契约 | 模型 JSON、模板和代码持续漂移 | 要求 folder-map 与六个 JSON Schema 为唯一事实源 |
| hash、文件名和重试语义不完整 | 更新文件、同内容异路径和重试产生重复卡片 | 定义 source/run/card 身份、稳定 fingerprint 和 upsert |
| 状态机缺少持久化检查点 | 重启后重新调用 API 或永久卡住 | 增加完整状态、远程任务 ID、lease、恢复和阶段重试 |
| 没有旧插件迁移方案 | 新代码保留旧路由/本地 fallback，测试互相冲突 | 增加 settings v3、模块迁移矩阵和旧数据处理策略 |

## P1 重要项

| 缺口 | v1.1 处理 |
|---|---|
| 审核仍可能退化为逐卡操作 | 按异常类型分组并提供批量批准、修正、重生和丢弃 |
| 进度只有阶段名称 | 增加远程页数、AI章节、累计卡片、耗时、暂停和取消 |
| API Key/外发风险描述不完整 | 明示 data.json 非加密、外发确认、日志脱敏和缓存敏感性 |
| API 限流、配额和成本不可见 | 增加并发、退避、重试上限和调用量展示 |
| “清空缓存”语义危险 | 拆分中间缓存、异常草稿和处理记录三种操作 |
| 双向关联可能被误解为修改两边正文 | 明确 wikilink 自动产生 backlink，关系候选最多 20 条并需语义复核 |
| MOC 与固定目录关系不够明确 | MOC 只查询标签，不移动卡片，统一放入 `_索引` |
| 缺少产品成功指标和发布门槛 | 增加零乱码、零重复、100%来源、恢复、28类冒烟等指标 |

## 对当前代码的复核

当前插件仍存在以下与 v1.1 冲突的实现：

- `external-pdf.js` 和设置页仍提供本地 MinerU、PaddleOCR、Python/pdfplumber。
- `ai-provider.js` 仍把分段原文直接生成卡片，并支持 OpenAI/Ollama。
- `routing.js` 仍按 Category/TagL1/TagL2 创建目录。
- `cards.js`、`tags.js` 和 `moc.js` 仍使用 Map_Index 参与旧流程。
- `task.js` 仍使用旧状态、0.82 阈值和旧解析顺序。
- 现有 routing、MOC、PDF fallback 等测试正在固定旧行为，必须重写而不是保留兼容。

这些差异已经逐模块写入 PRD 第 13 节，实施顺序和发布门槛也已确定。

## 就绪判断

- 产品决策：无阻塞未决项。
- 外部服务：MiniMax、MinerU 必需；PaddleOCR 为 PDF/图片补盲依赖。
- 首批工程产物：`folder-map.json`、六个 JSON Schema、settings v3 迁移测试。
- 结论：PRD 可以支撑修正插件，但不代表当前插件已经符合 PRD；必须按第 13.3 节顺序重构并通过第 12 节验收。
