# Obsidian 工程知识切片插件 PRD

## 版本记录

| 版本 | 日期 | 说明 |
|---|---|---|
| 1.0 | 2026-07-10 | 确立固定目录、云端解析、类型化总结和知识原子化工作流 |
| 1.1 | 2026-07-10 | 补齐用户目标、能力矩阵、长文档编排、数据契约、任务恢复、批量审核、迁移和验收要求，可作为插件重构依据 |

## 1. 产品定义

### 1.1 产品目标

将招投标和工程业务资料经过“导入、文档解析、类型化总结、知识原子化”四段处理，生成结构统一、来源可追溯、双向关联充分、可直接在 Obsidian 中检索和复用的中文知识卡片。

### 1.2 核心原则

1. 原始文件不可被插件改写。
2. MinerU/PaddleOCR 只负责把不同格式转换为大模型可稳定理解的 Markdown/结构化文本。
3. MiniMax M3 的第一类职责是类型判定和类型化总结，第二类职责是知识原子拆分与卡片生成；实际 HTTP 请求数由文档章节和长度决定。
4. 物理输出目录由固定业务分类决定，不由 MOC、Category、TagL1、TagL2 动态决定。
5. `Category / TagL1 / TagL2` 保留为检索元数据，不参与文件落盘路由。
6. 每张知识卡片必须链接原始文件和上游结构化总结。
7. 双向链接必须有真实语义关系，不为了增加图谱密度制造虚假链接。
8. 不得输出空洞总结、空白模板占位、无法从来源验证的事实。

### 1.3 本期边界

本期支持：PDF、DOC/DOCX、PPT/PPTX、XLS/XLSX、Markdown、TXT、HTML、EML、常见图片。音频、视频和 Outlook MSG 不进入处理队列，待后续接入 ASR、关键帧/OCR 或专用格式服务。

外部 LLM/Wiki、RAG 服务和团队同步仍为后续方向，本期只生成本地 Obsidian Markdown。

### 1.4 目标用户与核心任务

主要用户是只使用中文界面的营业、投标、设计、施工、项目管理、采购、合约、运维人员。用户不应理解 OCR、提示词、JSON Schema 或向量检索才能完成工作。

核心任务：

1. 把文件放入指定源文件夹或手动选择文件后，一次启动完整处理。
2. 随时看清当前文件处于上传、远程解析、结构化总结、知识原子化、校验还是写入阶段。
3. 自动获得多张有事实、有证据、有出处、有关系的中文知识卡片。
4. 只集中处理确实存在歧义的异常，不逐张确认正常卡片。
5. 能解释文件为何失败、为何跳过、为何进入审核，以及如何恢复。

### 1.5 产品成功指标

发布验收指标：

- 乱码或二进制碎片进入正式卡片的数量为 0。
- 正式卡片的源文件链接、证据定位和 `card_id` 完整率为 100%。
- 同一处理版本对同一来源重复执行，重复卡片率为 0。
- 28 个目录类型的路由合法率为 100%。
- 所有失败任务均显示阶段、原因和可执行的下一步。
- 插件重启后，已完成阶段不重复调用 API，处理中任务可恢复或明确失败。

试运行观察指标，不作为单个文件的硬性数量目标：自动入库率、异常审核率、每份文件有效知识原子数、人工驳回率、单文件耗时、API 调用次数和失败率。禁止为了提高“卡片数”制造低价值内容。

### 1.6 平台与非目标

- 本期仅支持 Obsidian Desktop，最低版本 `1.5.0`；移动端不承诺远程上传、长任务恢复和大文件处理。
- 本期不训练模型、不建设外部 Wiki/RAG、不自动修改源文件、不自动删除旧知识卡片。
- 本期不以成熟插件作为强依赖；Dataview、Bases、Tag Wrangler、QuickAdd、Templater 等仅作为可选增强。

### 2.4 运行时唯一事实源

插件运行时只读取库内组件包，不依赖 Downloads 或开发机绝对路径：

```text

├── folder-map.json
├── Tag_Library.md
├── 模板/

└── schemas/
```

- `folder-map.json` 是 28 个固定目录及提示词映射的唯一机器可读来源。
- `Tag_Library.md` 是标签白名单来源。
- `Type Mapping.md` 是静态/动态卡片类型来源。
- `schemas/*.schema.json` 是各阶段输入输出的校验来源。
- PRD 中的表格和提示词 frontmatter 是说明与校验副本，不得在代码中再维护第三套硬编码映射。
- 插件首次启动或升级时验证组件包版本并创建缺失的固定目录；组件包无效时禁止开始处理并列出缺失文件。

## 3. 四段式处理工作流

### 3.1 阶段一：文件导入

触发方式：

- 选择文件并指定“招投标”或“业务库”。
- 扫描输入目录并自动处理新增文件。
- 在 Obsidian 文件菜单中选择“用工程知识切片处理”。

导入任务至少记录：`task_id`、`run_id`、源文件路径、源文件别名、SHA-256、文件类型、库类型、pipeline/prompt/schema 版本、创建和更新时间、当前阶段、远程任务 ID、重试次数、中间产物路径、写入卡片 ID、错误历史和进度快照。

标准状态：

```text
discovered -> queued -> parsing -> parsed -> classifying -> classified
-> summarizing -> summarized -> atomizing -> validating -> writing -> written
```

旁路状态：`paused / cancelled / failed / needs_review / skipped / unsupported`。状态、阶段产物和远程任务 ID 每次变化后立即持久化，Obsidian 重启后从最后一个完整检查点继续，不能从头重复计费。

幂等与版本规则：

- `source_identity = library + source_hash`。同一库内相同内容只处理一次；不同路径记录为来源别名。
- 同一路径内容变化后生成新 source version，不覆盖旧卡片。
- `run_identity = source_identity + pipeline_version + prompt_bundle_version + schema_version`。版本变化后只提示“可重新处理”，不自动批量重跑历史资料。
- 每张卡片使用稳定 `card_id`；重试采用 upsert，不得通过文件名追加 `(1)` 产生重复卡片。
- 自动队列只消费 `queued`，失败任务仅由明确的“重试失败任务”操作重新入队；同一任务使用 lease 锁防止并发和重入。

### 3.2 阶段二：文档解析 API

#### 3.2.1 格式与解析器矩阵

| 输入格式 | 主处理方式 | 补盲方式 | 说明 |
|---|---|---|---|
| PDF | MinerU 精准解析 API | PaddleOCR API | 扫描、乱码、页面缺失或版面质量不合格时补盲 |
| PNG/JPG/JPEG/WEBP/GIF/BMP/TIF/TIFF | MinerU 精准解析 API | PaddleOCR API | 输出 Markdown、页面与图片资源 |
| DOC/DOCX、PPT/PPTX、XLS/XLSX | MinerU 精准解析 API | 无 | PaddleOCR API 不承担 Office 原文件解析；MinerU 失败则任务失败 |
| HTML | MinerU `MinerU-HTML` | 无 | 保留标题层级、表格和正文链接 |
| Markdown、TXT | 插件标准文本规范化 | 无 | 只做 BOM/编码识别、换行规范化和可读性校验，不属于解析失败后的本地兜底 |
| EML | 标准 MIME 解析器 | 附件按自身格式建立子任务 | 提取主题、发件人、收件人、日期、正文和附件引用；不支持 MSG |

MinerU 精准解析 API 当前限制按官方能力预检：单文件不超过 200 MB、文档不超过 200 页；超限文件不上传，任务直接说明需要拆分。PaddleOCR 仅接收其 API 支持的 PDF/图片类型。API 能力变化必须通过适配器和协议测试更新，不允许静默降级。

#### 3.2.2 远程解析策略

默认顺序：

1. MinerU 精准解析 API。
2. Markdown 质量校验。
3. 质量不合格时调用 PaddleOCR API 补盲。

MinerU 负责复杂版面、标题层级、表格、公式和多栏顺序；PaddleOCR 重点处理扫描件、图片型 PDF、旋转页面和 MinerU 低质量页面。两者不无条件双跑。

本产品不提供本地解析兜底。MinerU API 与 PaddleOCR API 均不可用、失败或超时时，任务进入 `failed`，保留脱敏错误和重试入口，不得使用浏览器直接读取二进制、简单字符串解码或本地 Python/PDF 库生成低质量文本。

统一解析包：

```json
{
  "source_path": "06-知识库/源文件/招投标/example.pdf",
  "source_hash": "sha256",
  "source_type": "pdf",
  "parser": "mineru-api",
  "language": "zh|en|ja|mixed",
  "markdown": "# ...",
  "pages": [{"page": 1, "text": "..."}],
  "images": [],
  "quality": {"readable": true, "score": 0.96}
}
```

若全部解析器失败或结果仍为乱码，禁止进入 MiniMax，必须保留各解析器原因并允许重试。

质量校验不得只看文本长度。至少检查：远程成功状态、预期/实际页数、有效字符比例、替换字符和控制字符比例、标题层级、表格结构、连续重复文本、异常脚本混杂、空白页比例。解析包保存 API 名称、模型、远程任务 ID、耗时和质量分项。

### 3.3 阶段三：MiniMax 类型判定与结构化总结

阶段三包含两项职责：

1. 使用 `提示词/00-类型判定.md` 判定库类型和固定目录类型。
2. 加载对应类型提示词，输出该类型的结构化总结 JSON。

“类型判定与结构化总结”是产品逻辑阶段，不等于固定一次 HTTP 请求。短文档可分别用一次请求完成分类和总结；长文档必须按标题、页码和表格边界切分，再使用 map/reduce 汇总。禁止按固定字符数截断后直接拼接卡片。

长文档编排：

1. 生成文档清单：标题层级、页码范围、表格和附件索引。
2. 分类请求只读取文件元数据、目录清单和代表性片段，返回前两名候选及证据。
3. 按章节生成带 `chunk_id`、页码范围和少量重叠上下文的局部总结。
4. 合并局部总结，消除重复并检查章节覆盖率，形成唯一结构化总结。
5. 任一章节遗漏、超出最大请求数或汇总失败时，整个总结阶段失败，不允许用前半份内容继续入库。

以下会议纪要提示词仅作为类型化总结的设计参考，不是固定字段清单，也不要求逐字写入最终提示词：

> 你是一名资深工程项目管理专家。请处理会议录音转写或会议纪要，剔除寒暄、过程噪音和非业务信息；根据原文实际内容整理会议背景、讨论问题、各方观点、已确认方案、未决事项、行动项等有价值的信息。专业术语应准确，所有结论应可回溯到输入内容。

实际会议提示词可根据文件内容扩展议程、背景资料、方案对比、约束条件、风险、决策依据、责任分工、截止日期和后续影响；来源中不存在的栏目应省略，不得为了满足模板补写空项或伪造结论。其他文件类型同样以对应目录提示词定义结构，不机械套用会议字段。

通用结构化总结外壳：

```json
{
  "document_type": "meeting-minutes",
  "library": "bid|business",
  "folder_type": "10-内部会议与评审",
  "output_folder": "06-知识库/wiki/招投标/10-内部会议与评审",
  "title": "...",
  "project": "...",
  "client": "...",
  "summary": {},
  "source_evidence": [{"page": 1, "excerpt": "..."}],
  "link_candidates": [{"title": "...", "relation": "supports|decides|depends-on|contradicts|same-project"}],
  "model_confidence": 0.0
}
```

各类型的 `summary` 字段不同，必须由对应提示词定义。严禁用一套会议总结字段套所有文件。

### 3.4 阶段四：MiniMax 知识原子化与卡片生成

第二次 MiniMax 调用读取结构化总结、Type Mapping、Tag Library、静态/动态模板和候选链接，输出严格 JSON；插件负责渲染 Markdown，不直接信任模型自由输出的 Markdown。

核心提示词要求：

> 你是一名知识管理专家。将结构化总结拆解为多个独立知识原子。每个独立决策、要求、参数、风险、问题、行动项、方法或可复用事实各生成一张卡片。依据 Type Mapping 选择静态信息卡片或动态事件卡片。Category、TagL1、TagL2 只能使用 Tag Library 已有值。所有卡片必须包含来源证据、源文件 wikilink 和上游总结 wikilink。只有存在明确语义关系时才输出 related 链接。零噪音输出严格 JSON。

原子化规则：

- 一张卡片只表达一个可独立引用的知识点。
- “召开了会议”“讨论了优化”“提升效率”等空泛表述不得成卡。
- 同一决策的背景、争议与结论可在一张动态卡片中形成闭环。
- 不同参数、条款、风险或行动项分别成卡。
- 相同来源、标题和证据指纹的卡片去重。
- 所有用户可见正文统一为简体中文，原文证据可附保留原语言。

结构化总结按主题/章节分组送入原子化阶段，最后进行全局合并：相同事实合并，冲突事实分别保留并建立 `contradicts` 候选关系。原子化前后必须核对总结中的决策、要求、参数、风险、问题和行动项覆盖率；不能因为达到卡片数量或请求上限而静默停止。

### 3.5 可信度计算与硬性门槛

`confidence` 是插件根据可验证结果计算的综合分，不直接采用 MiniMax 返回的自评小数。MiniMax 的 `model_confidence` 只作为诊断信息，最多参与类型判定分项，不能单独决定自动入库。

综合分计算：

```text
confidence = 0.25P + 0.15T + 0.35E + 0.15S + 0.10A
```

| 分项 | 权重 | 计算依据 |
|---|---:|---|
| `P` 解析质量 | 25% | API 返回状态、页面完整率、有效字符比例、乱码/控制字符比例、Markdown 标题和表格结构；解析不合格时不进入 MiniMax |
| `T` 类型判定 | 15% | 类型是否属于固定白名单、第一候选与第二候选差距、分类理由能否对应正文证据、输入库与判定库是否一致 |
| `E` 证据支持 | 35% | 卡片关键结论中有来源定位的比例、证据摘录能否在解析 Markdown 中回查、数字/日期/主体是否与证据一致 |
| `S` 结构合法性 | 15% | JSON Schema、必填字段、固定目录、Type Mapping、Tag Library 和 wikilink 格式是否通过校验 |
| `A` 原子质量 | 10% | 是否单一知识点、是否具有独立复用价值、是否为空泛表述、是否与现有或本批卡片重复 |

硬性门槛优先于加权总分：

- `P < 0.70`：解析失败，停止后续 AI 调用。
- 来源文件或来源定位缺失：不得入库。
- 关键结论的证据摘录无法在解析文本中回查：`confidence` 最高限制为 `0.59`。
- 固定目录、Type Mapping 或标签不在白名单：`confidence` 最高限制为 `0.69`。
- 检测到伪造数字、日期、主体或结论：拒绝该卡片并记录原因，不进入审核台等待人工补真。

默认决策区间：

- `confidence >= 0.85` 且无硬性错误：自动入库。
- `0.70 <= confidence < 0.85`：按批次进入异常审核台。
- `confidence < 0.70`：自动重新生成一次；仍不达标则标记失败。

审核台必须显示总分、五个分项、触发的硬性门槛和可定位的失败原因，不能只显示一个无法解释的小数。

## 4. 类型与固定路由

### 4.1 招投标类型

| 固定目录 | 结构化总结重点 | 提示词文件 |

## 5. Type Mapping

`模板/Type Mapping.md` 是卡片类型的唯一映射源：

- 动态事件：发生了什么、做出了什么决定、谁要做什么、状态如何变化。
- 静态信息：是什么、有什么参数/规则、适用边界是什么、如何复用。

目录类型与卡片类型是两个维度。一个招标文件可产生静态条款卡、动态风险卡和行动项卡，但所有卡片仍落入 `02-招标文件解读` 固定目录。

## 6. 卡片模型

### 6.1 通用 Frontmatter

```yaml
---
title: ""
card_id: ""
atom_fingerprint: ""
card_kind: static|event
Info_Type: ""
Event_Type: ""
library: bid|business
folder_type: ""
output_folder: ""
project: ""
client: ""
stage: ""
status: draft|confirmed|superseded
Category: ""
TagL1: ""
TagL2: ""
created: YYYY-MM-DD
updated: YYYY-MM-DD
source_file: ""
source_link: "[[06-知识库/源文件/...]]"
source_hash: ""
source_page: ""
parent_summary: "[[...]]"
related: []
aliases: []
tags: []
confidence: 0.0
confidence_components: {}
schema_version: "1.1"
pipeline_version: ""
prompt_bundle_version: ""
---
```

`card_id = source_hash 前 12 位 + atom_fingerprint 前 12 位`。`atom_fingerprint` 由规范化知识主语、谓语、关键对象/参数、证据定位和卡片类型计算；标题变化不改变卡片身份。

### 6.2 来源要求

- `source_link` 必填并使用 vault 内 wikilink。
- 有页码时写 `source_page`，正文可使用 `![[源文件.pdf#page=12]]`。
- `parent_summary` 必填，链接阶段三的结构化总结。
- `source_excerpt` 在正文“来源证据”中显示，必须是最短可验证摘录。

### 6.3 阶段数据契约

组件包必须提供并由插件在运行时强校验：

```text
schemas/
├── task.schema.json
├── parse-package.schema.json
├── classification.schema.json
├── structured-summary.schema.json
├── knowledge-atoms.schema.json
└── card.schema.json
```

- MiniMax 响应必须先解析为 JSON，再通过对应 Schema；不得从自由 Markdown 中用正则猜字段。
- Schema 失败时最多使用“带校验错误的修复提示词”重试一次；仍失败则进入 `failed`。
- 模型输出的 `model_confidence`、目录、标签、链接和日期均视为不可信输入，必须再校验。
- 每个阶段产物写入 `schema_version / pipeline_version / prompt_bundle_version`，避免新旧缓存混用。

### 6.4 模板渲染与写入

- 插件内置确定性模板渲染器，支持转义、列表和可选区块；Templater/Templates 不作为运行依赖。
- 空值对应的 frontmatter 属性和正文小节不渲染，禁止留下 `{{placeholder}}`、“待补充”或空标题。
- 文件名使用经过清洗的中文标题；身份和去重只依赖 `card_id`，不依赖文件名。
- 写入前先完成整批校验和稳定 ID 计算；单卡使用临时文件写入后原子替换。
- 已存在同 `card_id` 且来源版本相同则更新，不新建副本；来源版本变化则新建版本并建立 `supersedes` 关系，旧卡片标记 `superseded`，不自动删除。
- 源文件始终留在输入目录，不移动到 `_processed`。处理状态只记录在任务账本。

## 7. 双向关联和知识图谱

### 7.1 最低关联要求

每张正式卡片至少具备：

1. 源文件链接。
2. 上游结构化总结链接。
3. 所属项目主页或客户档案链接（能可靠识别时）。

### 7.2 语义关系

`related` 使用 wikilink 列表，关系类型记录在正文：

- `supports`：证据支持。
- `decides`：会议/审批决定。
- `depends-on`：依赖前置条件。
- `contradicts`：存在冲突。
- `mitigates`：措施应对风险。
- `supersedes`：新版本替代旧版本。
- `same-project`：同项目阶段关系。
- `derived-from`：从总结或方法派生。

插件应先在现有 wiki 中按项目、客户、实体、标题关键词和标签寻找候选，再由 MiniMax 判断关系。低置信度候选不自动写入。

候选检索分两级：先用确定性条件检索同项目、同客户、同实体和同标签卡片，再将最多 20 个候选的标题、摘要和关系类型交给 MiniMax 复核。只有关系类型在白名单、双方实体能对齐且有明确理由时写入。Obsidian 会根据 wikilink 自动形成反向链接，插件不为“制造双向”修改被链接卡片正文。

### 7.3 MOC 与索引

- `Category / TagL1 / TagL2` 继续作为 MOC 主索引三元组，但不参与物理目录路由。
- MOC 页面统一放入 `06-知识库/wiki/_索引/`，使用 Dataview/Bases 查询固定目录下的卡片。
- 插件可创建缺失的 MOC 页面或更新索引数据，不得因 MOC 变化移动卡片。
- 无 Dataview/Bases 时，普通 Obsidian 搜索、属性和反向链接仍可使用；插件不得加载失败。

MOC、Bases、Dataview 和 Graph View 是索引展示层，不再决定物理文件夹。可按 `library`、`folder_type`、`project`、`Info_Type/Event_Type` 和标签动态生成导航。

## 9. 审核和自动入库

默认采用批量自动入库：字段、固定目录、来源、标签和置信度全部通过时直接写入；以下内容进入审核台：

- 类型判定冲突、综合分低于自动入库阈值或触发硬性门槛。
- 固定目录无法确定。
- 标签不在 Tag Library。
- 来源证据不足。
- 关联候选冲突。
- 涉密文件被设置为强制审核。

审核以任务批次和异常类型聚合，不要求逐张确认所有可信卡片。

审核台必须支持：

- 按来源文件、失败阶段、非法标签、低证据分、路由冲突和关联冲突分组。
- “批准本组”“应用同一标签修正”“修改固定目录”“重新生成本组”“丢弃本组”。
- 同时显示解析文本/结构化总结、知识卡片、证据原文和五项可信度分数。
- 批量操作前显示影响数量；批量批准仍必须通过 Schema、来源和标签硬性校验。
- 可信卡片已自动入库时，重试异常卡片不得再次生成或覆盖无关卡片。

任务内部分卡片通过、部分卡片异常时，任务状态为 `needs_review`，并分别记录 `written_card_ids` 与 `review_atom_ids`；全部异常解决后转为 `written`。

## 10. 界面与设置

### 10.1 主界面

插件页面固定为 Obsidian 工作区可用高度，整体不出现纵向滚动；只有审核工作台内部滚动。纵向顺序固定：

1. 总览：扫描并自动处理、继续队列、重试失败任务、打开设置及状态计数。
2. 处理概览：当前文件、当前阶段、远程解析器、页数进度、AI 章节/分段进度、累计卡片数、耗时、暂停/取消。
3. 审核工作台：只显示异常批次，文本字号比普通正文小两级。
4. 底部固定信息：输入目录、输出目录、当前 pipeline/prompt/schema 版本。

“已跳过”必须可展开查看原因统计，例如内容 hash 已处理、格式不支持、超出 API 限制、用户取消；主界面不显示冗长文件列表。

### 10.2 设置项

- 输入目录：招投标、业务库。
- 两套固定输出根目录。
- MinerU/PaddleOCR API Token、端点、模型、超时和外部上传开关。
- MiniMax 国内版 API Key、模型、端点和分段参数。
- 自动处理开关、自动入库阈值、重新生成次数和敏感资料审核策略；五项评分权重默认固定，高级设置中才允许调整。
- 提示词目录、Tag Library、Type Mapping 和模板路径。
- 缓存清理、失败重试和任务日志。

设置页只显示 MiniMax 国内版、MinerU API 和 PaddleOCR API，不显示 OpenAI、Ollama、本地 MinerU、Python、pdfplumber 或本地 PaddleOCR 设置。三个 API 分别提供“测试连接”；测试必须验证鉴权和最小可用响应，并显示脱敏错误。

### 10.3 首次启动就绪检查

开始处理前必须全部满足：MiniMax Key 可用、MinerU Key 可用、外部上传已确认、组件包与 28 个目录映射通过校验。PaddleOCR Key 缺失时允许启动，但明确提示 PDF/图片补盲不可用；一旦 MinerU 质量不合格，该任务失败而不是继续生成卡片。

## 11. 安全、配额与维护

### 11.1 安全与隐私

- API Key 只保存在插件 `data.json`，不写入 Markdown 和日志。Obsidian 插件设置文件不是加密保险箱，设置页必须明确提示这一点；日志和错误界面统一脱敏。
- 外部上传默认关闭，由用户确认公司保密和数据外发要求后开启。关闭时云端解析任务不得开始，并显示就绪检查原因。
- 源文件只读，插件不得改名、移动、覆盖或删除。
- 解析缓存和结构化总结可能包含敏感正文，默认仅保存在当前 vault；不进入正式索引，不被日志复制。
- 远程下载 URL、任务 ID、HTTP 响应和模型响应不得包含 Token；调试日志也必须经过同一脱敏函数。

### 11.2 限流、重试与成本可见性

- 默认同时只处理 1 个源文件；同一服务的并发数和轮询间隔可配置，但不得突破服务端限制。
- HTTP 429/5xx、网络中断和远程 `pending/running` 使用带抖动的指数退避，优先遵循 `Retry-After`；单阶段默认最多自动重试 3 次。
- 鉴权失败、格式不支持、超出大小/页数和 Schema 错误不做无意义重试。
- 队列显示页数、解析服务、MiniMax 请求数、输入字符/Token 估算和耗时；不承诺无法从服务端取得的精确费用。
- 用户可暂停队列、取消尚未提交的任务；已提交远程任务如 API 不支持取消，则停止轮询并标记取消，不继续 MiniMax。

### 11.3 缓存与重跑

设置页分开提供三个动作，不能合并成一个含义不清的“清空”：

1. **清理中间缓存**：删除 parsed、summaries 和过期日志；保留任务账本、源文件、审核项和正式卡片。
2. **清空异常草稿**：只删除用户选中的审核批次，不修改正式卡片。
3. **重置处理记录**：清空任务账本，使源文件可重新扫描；必须二次确认并明确提示可能产生新版本。源文件和正式卡片仍不删除，稳定 `card_id` 负责防重。

任务级提供“按当前版本重新处理”和“回滚本次写入”。回滚只处理本次 `run_id` 创建或更新的文件，并保留操作日志。

## 12. 验收与发布门槛

### 12.1 功能验收

1. 两个输入目录均可选择导入和自动扫描；库类型来源清楚。
2. PDF、Word、Excel、PPT、HTML、图片能通过 MinerU 生成可读 Markdown；低质量 PDF/图片按规则转 PaddleOCR；Markdown/TXT/EML 使用各自标准读取器。
3. 乱码、二进制碎片、缺页结果和未完整处理的长文档不得进入 MiniMax 原子化阶段。
4. 28 个固定目录类型都有独立提示词，并由 `folder-map.json` 唯一路由。
5. 会议、招标文件、报价、风险、客户和案例使用各自结构；来源不存在的栏目不生成。
6. 一个文件能产生全部有价值知识原子，不按固定卡片数量截断，也不生成空泛卡片。
7. 每张正式卡片有稳定 `card_id`、源文件、结构化总结、证据定位、版本和可信度分项。
8. `Category / TagL1 / TagL2` 只用于标签和 MOC 查询，不参与文件夹路由。
9. 可信卡片自动入库；异常按批次审核并支持批量修正。
10. 同一 run 重试不重复写卡；源文件新版本产生可追踪的新卡版本和 `supersedes` 关系。
11. 插件重启、网络中断和 Obsidian 重载后能从检查点恢复，不重复远程解析。
12. 所有用户界面和生成内容为中文；原文证据可保留中文、英文或日文。

### 12.2 必测样本矩阵

| 维度 | 必测样本 |
|---|---|
| 语言 | 中文、英文、日文、中英日混合 |
| PDF | 文本型、扫描型、双栏、旋转、表格、公式、盖章、100+页长文档 |
| Office | DOCX、PPTX、XLSX，含表格、图片和多工作表/多页 |
| 其他 | Markdown、UTF-8/UTF-16/GB18030/Shift-JIS TXT、HTML、含附件 EML、常见图片 |
| 业务类型 | 28 类至少各 1 份；会议、招标、报价、风险、客户、失败复盘各有复杂样本 |

### 14.2 代码与组件依赖

- Obsidian API，Desktop `1.5.0+`。
- JSON Schema 校验器，建议 `ajv`，打包进插件。
- 确定性模板渲染器，建议 `mustache`，打包进插件。
- EML MIME 解析器，建议使用浏览器兼容的 `postal-mime`。
- 文本编码检测库配合 Chromium `TextDecoder`，覆盖 UTF-8、UTF-16、GB18030 和 Shift-JIS。
- 组件包：`folder-map.json`、31 个提示词、Tag Library、Type Mapping、2 个模板、6 个 JSON Schema。

成熟 Obsidian 插件全部为可选增强，缺失时不影响主流程。运行时不依赖 `C:/Users/.../Downloads`、Python、命令行工具或 `kepano/obsidian-skills` 仓库。

## 15. 产品复核结论

PRD v1.0 的产品方向正确，但不足以无歧义指导插件重构，主要缺少格式/API边界、长文档策略、机器可校验契约、稳定身份、恢复与重试、批量审核操作、缓存语义、迁移和发布门槛。v1.1 已补齐这些产品决策。

本 PRD 现在可以作为插件重构的产品与验收依据。开始编码前仍需按 13.3 第 1 项生成 `folder-map.json` 和六个 JSON Schema；它们是由本 PRD 明确定义的首批工程产物，不再需要额外产品决策。
